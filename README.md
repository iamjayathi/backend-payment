# Payment Processing System

A TypeScript backend assignment that simulates asynchronous payment processing with idempotency, background jobs, retries, and gateway failures.

## What It Does

The API creates a payment record immediately, then processes it in the background through a Redis queue. A worker picks up the job, calls a simulated payment gateway, and updates the payment status in PostgreSQL.

```text
Client
  -> Express API
  -> PostgreSQL payment record
  -> BullMQ job in Redis
  -> Worker
  -> Gateway simulator
  -> PostgreSQL status update
```

## Tech Stack

- Node.js + TypeScript
- Express
- PostgreSQL
- Redis + BullMQ
- Winston logging
- Swagger API docs

## Payment Flow

```text
pending -> processing -> success
                    -> failed
```

The gateway simulator randomly returns success, non-retriable failure, or timeout. Timeout and retriable failures are retried by BullMQ with exponential backoff.

## Main Design Choices

| Concern | Implementation |
|---|---|
| Idempotency | `idempotency_key` has a unique DB index. Duplicate create requests return the existing payment. |
| Async processing | API enqueues a BullMQ job and returns the payment immediately. |
| Concurrency | Worker uses an atomic `pending -> processing` DB update so only one job can process a payment. |
| Retries | BullMQ retries timeout/retriable gateway errors. |
| Webhooks | Gateway callbacks are stored idempotently and applied only if the payment is not already terminal. |
| Observability | Logs show payment creation, queue pickup, gateway result, retry, and final status. |

## Setup

### 1. Install Dependencies

```bash
npm install
```

### 2. Start PostgreSQL and Redis

Using Docker:

```bash
docker compose up -d
```

Or use your own local PostgreSQL and Redis, then update `.env`.

### 3. Configure Environment

```bash
cp .env.example .env
```

Default local values:

```env
PORT=3000
DATABASE_URL=postgres://postgres:postgres@localhost:5432/payments_db
REDIS_URL=redis://localhost:6379
MAX_RETRIES=3
RETRY_DELAY_BASE_MS=1000
WEBHOOK_SECRET=your-webhook-secret-here
```

### 4. Run Migration

```bash
npm run db:migrate
```

### 5. Start API

```bash
npm run dev
```

### 6. Start Worker

Open another terminal:

```bash
npm run worker
```

## API Routes

Base URL:

```text
http://localhost:3000/api
```

### Health

```http
GET /api/health
```

### Create Payment

```http
POST /api/payments
Content-Type: application/json
```

```json
{
  "idempotency_key": "order_1001",
  "amount": 99.99,
  "currency": "USD",
  "metadata": {
    "order_id": "1001"
  }
}
```

### Get Payment

```http
GET /api/payments/:id
```

### Webhook

```http
POST /api/webhooks/payment
Content-Type: application/json
```

```json
{
  "transaction_id": "txn_abc123",
  "payment_id": "00000000-0000-4000-8000-000000000001",
  "status": "success",
  "timestamp": "2026-05-06T10:00:00Z"
}
```

`payment_id` is optional. It allows the simulator to handle early callbacks before a transaction id has been attached to the payment. Duplicate or conflicting callbacks are stored, but terminal payments are not overwritten.

### Dead Letter Queue

```http
GET /api/admin/dlq?limit=50
```

